# Database migration how-to

This project uses Flask-Migrate / Alembic for schema changes.

This is the safest repeatable workflow for making a database change locally and applying it in production.

## Golden rules

1. **Never edit production tables by hand.**
2. **Never rely on `db.create_all()` for production schema changes.** Use migrations.
3. **Every model change and migration should ship together in one commit.**
4. **Always test the migration on a copy of real-ish data before production.**
5. **Always take a production backup immediately before running `upgrade`.**
6. **For SQLite, backup/restore is your primary rollback plan.** Do not depend on downgrade working under pressure.

---

## Quick checklist

When changing the schema:

- Update the SQLAlchemy models.
- Generate a migration.
- Review and fix the migration file manually.
- Run the migration locally.
- Run the test suite.
- Test on a copy of production data.
- Commit code + migration together.
- Backup production.
- Deploy code.
- Run `flask db upgrade` in production.
- Smoke test the app.

---

## 1) Make the model change locally

Example changes:

- add a column
- rename a column
- create a table
- add an index
- change nullable/default rules

Update the relevant model(s) in [app/models.py](app/models.py) or other model modules first.

### Important

A migration should describe the schema change explicitly. Autogenerate helps, but it is **not** perfect.

Common things Alembic may need help with:

- renames show up as drop + add
- SQLite batch operations may need adjustment
- data backfills are usually manual
- indexes / constraints should be reviewed carefully

---

## 2) Generate a migration locally

From the project root:

### PowerShell

```powershell
$env:FLASK_APP = 'run.py'
venv\Scripts\python.exe -m flask db migrate -m "describe the change"
```

### cmd.exe

```bat
set FLASK_APP=run.py
venv\Scripts\python.exe -m flask db migrate -m "describe the change"
```

This creates a new file in [migrations/versions](migrations/versions).

---

## 3) Review the migration file before running it

Open the new migration in [migrations/versions](migrations/versions) and check:

- Are the table and column names correct?
- Did a rename get generated as drop/add instead of a true rename?
- Are `nullable` changes safe for existing rows?
- Do new non-null columns need a temporary default or backfill?
- Are indexes / foreign keys present if expected?
- Does `downgrade()` make sense?

### Required mindset

**Do not trust autogenerated migrations blindly.**

If existing rows would violate the new schema, add a data migration step inside `upgrade()`.

Example pattern:

1. add nullable column
2. backfill values
3. alter column to non-null

That is safer than adding a non-null column to a table that already has data.

---

## 4) Apply the migration locally

Run:

### PowerShell

```powershell
$env:FLASK_APP = 'run.py'
venv\Scripts\python.exe -m flask db upgrade
```

Then verify:

- the app starts
- the affected screens still load
- create/edit flows still work
- old data still reads correctly

If needed, inspect the current revision:

```powershell
$env:FLASK_APP = 'run.py'
venv\Scripts\python.exe -m flask db current
```

---

## 5) Run tests after the migration

From the project root:

```powershell
venv\Scripts\python.exe -m pytest
```

If the schema change affects a workflow, add or update tests before moving on.

---

## 6) Rehearse on a copy of production data

This is the step that prevents most migration disasters.

### For SQLite production databases

Make a copy of the production DB file and test the migration against the copy locally.

Recommended flow:

1. Copy the production DB file to a safe local temp location.
2. Point `DATABASE_URL` at that copy.
3. Run `flask db upgrade` against the copy.
4. Start the app against the upgraded copy.
5. Test the exact affected admin/user flows.

### Example PowerShell

```powershell
$env:FLASK_APP = 'run.py'
$env:DATABASE_URL = 'sqlite:///C:/temp/prod-copy.db'
venv\Scripts\python.exe -m flask db upgrade
```

If the migration fails here, fix it before production.

---

## 7) Commit code and migration together

A schema change is not complete unless the following ship together:

- model changes
- migration file
- tests
- any code that uses the new schema

Never deploy model changes without the matching migration.
Never deploy the migration without the code that expects it.

---

## 8) Production deployment plan

Use this order every time.

### A. Prepare

- Confirm the branch/commit is final.
- Confirm the migration was tested locally and on a prod-like copy.
- Identify the production DB location.
- Schedule a short maintenance window if the app is live.

### B. Backup production first

For SQLite, make a full file copy before anything else.

Example:

```powershell
Copy-Item C:\path\to\prod.db C:\path\to\backup\prod-before-2026-04-07.db
```

If production is Linux:

```bash
cp /path/to/prod.db /path/to/backup/prod-before-2026-04-07.db
```

Do not skip this.

### C. Stop writes if possible

Best practice is to stop the app briefly or put it into maintenance mode so no writes happen mid-migration.

This matters more for:

- column rewrites
- non-null changes
- data backfills
- table rebuilds

### D. Deploy code

Deploy the application code that includes:

- updated models
- new migration file
- related route/template/API changes

### E. Run the migration in production

Example:

```bash
export FLASK_APP=run.py
python -m flask db upgrade
```

Or with the venv path explicitly:

```bash
/path/to/venv/bin/python -m flask db upgrade
```

For Windows server:

```powershell
$env:FLASK_APP = 'run.py'
path\to\venv\Scripts\python.exe -m flask db upgrade
```

### F. Smoke test immediately

Check the exact impacted paths right away:

- app starts cleanly
- login works
- admin page loads
- changed create/edit flow works
- existing records still load

---

## 9) Rollback plan

### Best rollback for SQLite

Restore the DB backup and redeploy the last known-good code.

That is usually more reliable than trying to improvise a downgrade under time pressure.

### If using Alembic downgrade

Only use downgrade if you have already tested it and know it is safe.

Check the target revision:

```powershell
$env:FLASK_APP = 'run.py'
venv\Scripts\python.exe -m flask db history
```

Downgrade one step:

```powershell
$env:FLASK_APP = 'run.py'
venv\Scripts\python.exe -m flask db downgrade -1
```

But again: **for this project, a tested DB backup restore is the safer primary rollback plan.**

---

## 10) Safe patterns for risky changes

### Adding a non-null column to a table with existing rows

Safer sequence:

1. add the column as nullable
2. backfill data
3. make it non-null in a second step

### Renaming a column

Do not assume autogenerate will do the right thing.

Often safer:

1. add new column
2. copy data from old column to new column
3. ship code that reads the new column
4. later remove old column in a follow-up migration

### Large or risky data rewrites

Use the **expand / migrate / contract** approach:

1. **expand**: add new schema without breaking old code
2. **migrate**: backfill or dual-write data
3. **contract**: remove old schema later

This is the most reliable pattern for production changes.

---

## 11) What to avoid

Avoid these habits:

- editing the DB manually in production
- changing models without creating a migration
- running unreviewed autogenerated migrations
- making schema + big data rewrite + feature rollout all at once if avoidable
- skipping the backup because the change "looks small"
- assuming downgrade will save you later

---

## 12) Standard operating procedure for this project

Use this exact flow:

1. Change model code.
2. Generate migration.
3. Review migration file manually.
4. Run `flask db upgrade` locally.
5. Run tests.
6. Rehearse against a copy of production DB.
7. Commit code + migration together.
8. Backup production DB.
9. Stop app or maintenance mode if possible.
10. Deploy code.
11. Run `flask db upgrade` in production.
12. Smoke test.
13. Keep the backup until the release is proven stable.

## Final advice

If a migration changes existing data, treat it like code, not plumbing.

The safest production move is:

- rehearse on a copy
- backup before upgrade
- deploy code and migration together
- keep rollback simple

For this app, especially if production uses SQLite, **backup + test on a copy + `flask db upgrade`** is the rock-solid path.
